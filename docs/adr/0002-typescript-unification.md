# ADR-0002: TypeScript統一（Rust不採用）

## ステータス

採用 (Accepted)

## 日付

2026-02-28

## コンテキスト

Auto Kanbanは、個人利用向けのAIエージェント統合タスク管理アプリケーションである。バックエンド言語の選定にあたり、以下の点を検討した：

1. **言語の統一**: フロント・バックで同一言語を使用したい
2. **型共有の容易さ**: コード生成ツールなしで型を共有したい
3. **開発速度**: 高速な開発サイクルを実現したい
4. **シンプルな構成**: パッケージ構成を最小限に保ちたい

### 検討した選択肢

#### 選択肢1: Rust (Axum + SQLx)

**利点:**
- 高い実行性能
- メモリ安全性の保証

**欠点:**
- フロント・バック間の型共有にts-rs等のコード生成が必要
- 学習コストが高い
- コンパイル時間が長い
- 個人開発ではRustの安全性保証のメリットが薄い

#### 選択肢2: TypeScript統一（Bun）

**利点:**
- フロント・バックで同一言語
- tRPCによるゼロコスト型共有
- 開発サイクルが速い
- 学習コストが低い
- Bunによる高速実行・シングルバイナリ

**欠点:**
- Rustほどの実行性能は出ない
- ランタイムエラーのリスク（型チェックは静的のみ）

#### 選択肢3: Go

**利点:**
- シンプルな言語仕様
- シングルバイナリ生成
- 高速なコンパイル

**欠点:**
- フロントエンドとの型共有が困難
- ジェネリクスの表現力が低い

## 決定

**TypeScript統一（Bun）を採用する。**

## 根拠

1. **型共有のシームレス化**: tRPCにより、バックエンドで定義した型がフロントエンドで自動的に利用可能。コード生成ツール不要。

2. **開発速度**: 単一言語でフルスタック開発が可能。ホットリロードによる即時フィードバック。

3. **学習コスト**: TypeScriptの知識のみで全体を理解・修正可能。

4. **Bunによる性能担保**:
   - JITコンパイルによる高速実行
   - ビルトインSQLite (bun:sqlite)はbetter-sqlite3より3-6倍高速
   - シングルバイナリ生成でRustと同様の配布体験

5. **個人利用に最適**: 大規模チームでのRustの安全性メリットより、個人開発での素早いイテレーションを重視。

6. **シンプル化**: 10 crates → 2 packages (server, web) に削減。

## 結果

### ポジティブ

- tRPCによる型安全なAPI（コード生成不要）
- 単一言語による認知負荷の軽減
- 高速な開発サイクル（ホットリロード）
- フロント・バック間でのコード共有が可能（バリデーションスキーマ等）
- Bunによるシングルバイナリ配布

### ネガティブ

- CPU負荷の高い処理ではRustに劣る（ただし本アプリでは問題にならない）
- ランタイムでの型エラーの可能性（静的型チェックでカバー）

### パフォーマンス比較

| 操作 | Rust + Axum | Bun + Hono | 差異 |
|------|-------------|------------|------|
| HTTP リクエスト/秒 | ~100k | ~80k | 許容範囲 |
| メモリ使用量 | ~20MB | ~50MB | 許容範囲 |
| 起動時間 | ~10ms | ~50ms | 許容範囲 |

※ 個人利用のタスク管理アプリでは上記の差異は実用上問題にならない。
